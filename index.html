The contract ships the UI

dapp32 lets an Ethereum contract point to a pinned offchain CAM manifest that defines the dapp's screens, reads, inputs, and wallet actions. That interface can be rendered as a graphical UI or exposed as a minimal API for agents. Developers can run the open dapp32 viewer, embed it, or use a hosted viewer they trust instead of building a whole custom frontend.

Current CAM workflow

1
Host contract

Publishes camURI and camHash.

2
main.json

Declares namespaces, ABI resources, routes, and continuations.

3
ui.json

Declares a small portable UI node catalog.

4
Generic viewer

Loads, validates, renders, simulates, and hands writes to a wallet.

Why this matters

Smart contracts enforce dapp rules, but the workflow a human or agent follows is usually encoded in a custom website: what to display, what to collect from the user, what a button does, and what should happen after a transaction. Every team rebuilds that UI layer even when the contract already knows the relevant state and permissions.

dapp32 moves that workflow description into declarative manifest data published by the contract. A generic viewer validates the manifest, then renders a human UI or exposes the same flow as a barebones agent API before preparing wallet actions.

Approach

Contract-anchored manifests

A host contract points to a pinned CAM bundle. The app's screens, reads, and actions live in reviewable manifest data instead of a custom frontend codebase.

Routes before rendering

Reads are explicit route calls into contract views. Their outputs flow into a declarative UI catalog. Writes become prepared ABI calls for simulation and wallet review.

Small UI model

The UI resource describes portable nodes such as text, fields, addresses, status rows, NFT references, includes, and actions. Those nodes can become graphical controls or a minimal machine interface, not an app-specific runtime every project must rebuild.

Current evidence

  • The local bike NFT scenario deploys a real contract fixture, serves CAM resources, and runs in a graphical viewer.
  • Runtime packages load CAM hosts, verify resources, resolve contract namespaces, read routes, and prepare writes.
  • @cam/viewer provides a headless state machine that a terminal, web app, wallet, or agent can drive.
  • @cam/conformance catches authoring errors across manifests, ABI usage, resources, routes, UI, and typeflow.
  • The React viewer is deliberately thin: DOM and wallet interaction stay outside the protocol packages.
  • A generic integration-fuzz runner walks deployed CAM hosts to catch workflow failures that static checks miss.

Local demo

The current path is Docker-backed through the Makefile. Dependency installation is explicit and may take time. The GUI target starts a local Anvil chain, deploys the bike NFT fixture, serves CAM resources, and launches the React viewer through a local gateway. The target prints the URL to open in your browser. 

git clone https://github.com/numpde/dapp32.git
cd dapp32
make deps
make package-deps
make cam-conformance-check
make bike-nft-viewer-gui
# open the printed http://127.0.0.1:5173/... URL

What is in the repo

Protocol packages

@cam/protocol, @cam/core, @cam/screen, @cam/evm-viem, @cam/viewer, and @cam/conformance parse, validate, resolve, and test CAM bundles.

Reference app

cam-web is a thin renderer around the headless viewer. The browser owns DOM and wallet interaction; protocol logic stays in reusable packages.

Demo dapp

dapps/bike-nft exercises registration, lookup, metadata, missing/clear, retirement, local deployment, and CAM UI projection.

Adjacent ideas

These projects are useful reference points, but most solve a neighboring layer: naming, resource transport, signatures, transaction submission, or distribution. dapp32 is about the UI/workflow layer a contract can publish for a shared viewer or agent-facing interface.

TokenScript / ERC-5169 lets token contracts expose client-side resources, commonly for token-aware wallet experiences. It is useful precedent for contract-discovered UI material, but dapp32 needs a small declarative UI/workflow manifest a generic viewer can validate before rendering.

ERC-4804 web3:// is a URL scheme for reading resources through Ethereum contract calls. It tells a client how to fetch something, not what UI to render, which reads to run, which inputs to collect, or which wallet action to prepare.

CCIP Read / ERC-3668 is a contract-authorized offchain lookup pattern using OffchainLookup and callback verification. It helps a contract obtain or verify offchain data, but it does not define the app's screens, navigation, form inputs, or action grammar.

ENS contenthash points a name at content-addressed resources such as IPFS or Swarm content. It locates bytes, but it does not say whether those bytes are a safe UI manifest, how to validate them, or how they map to contract reads and wallet actions.

ERC-4337 account abstraction is an account-abstraction system built around smart accounts, UserOperations, bundlers, and paymasters. It changes how transactions are authorized, paid for, and submitted; it starts after the UI/workflow has already produced an action.

EIP-5792 wallet call batches is a wallet API for sending and tracking one or more calls, with wallet capability discovery. It is a good handoff target after a viewer prepares calls, but it does not define how the dapp publishes the UI flow that leads to those calls.

EIP-712 typed data is a standard for typed, human-readable structured data that wallets can present before signing. It describes a signing payload, not the app UI, route graph, or contract-backed flow that produces that payload.

ERC-681 transaction requests is a URI format for payment and transaction requests. It is useful for one action or one QR code; it is not a multi-screen dapp workflow with reads, inputs, validation, and follow-up actions.

EIP-1193 and EIP-6963 are browser/provider standards for talking to wallets and discovering multiple injected wallets. They are wallet plumbing; they do not define the app's screens, contract reads, input model, or prepared action semantics.

Solana Actions / Blinks are API-driven action descriptors that clients can render as buttons, links, and transaction flows in the Solana ecosystem. They prove the action-surface need, but the authority is an API endpoint; dapp32 anchors the UI/workflow pointer and validation boundary in an Ethereum contract.

Farcaster Mini Apps are web apps packaged for Farcaster clients, with social context, distribution, and client integration. They are a useful container for apps, but still assume an app runtime; dapp32 targets the smaller contract-defined UI/workflow layer underneath.

ERC-7572 contract metadata is a contract-level metadata URI pattern for describing a contract or collection. Metadata describes what a contract is; it does not define an interactive UI flow a shared viewer can execute step by step.
