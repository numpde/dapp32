Contract-defined workflows

Can contracts publish workflows generic clients can trust?

dapp32 is a research prototype for technical evaluators asking whether Ethereum contracts can describe app workflows directly. It uses contract-published, hash-pinned CAM bundles that generic viewers, wallets, and agents can validate, render, and use to prepare actions without trusting bespoke frontend JavaScript.

Current CAM workflow

1
Host contract

Publishes camURI and camHash.

2
main.json

Declares namespaces, ABI resources, routes, and continuations.

3
ui.json

Declares a small portable UI node catalog.

4
Generic viewer

Loads, validates, renders, simulates, and hands writes to a wallet.

Why this matters

Smart contracts enforce dapp rules, but the workflow a human or agent follows is usually encoded in a custom website: what to display, what to ask for, what a button does, and what should happen after a transaction.

That website becomes an unverified offchain interpreter of the contract-governed workflow. dapp32 asks how much of that workflow can instead be published by the contract as declarative, validated data for generic viewers, wallets, and agents.

Approach

Contract-anchored manifests

A host contract points to a CAM bundle. The root manifest and secondary ABI/UI resources are integrity checked before a client renders or prepares anything.

Routes before rendering

Reads are explicit route calls into contract views. Their outputs flow into a declarative UI catalog. Writes become prepared ABI calls for simulation and wallet review.

Small UI model

The UI resource describes portable nodes such as text, fields, addresses, status rows, NFT references, includes, and actions. It is data, not an arbitrary app runtime.

Current evidence

  • The local bike NFT scenario deploys a real contract fixture, serves CAM resources, and runs in a graphical viewer.
  • Runtime packages load CAM hosts, verify resources, resolve contract namespaces, read routes, and prepare writes.
  • @cam/viewer provides a headless state machine that a terminal, web app, wallet, or agent can drive.
  • @cam/conformance catches authoring errors across manifests, ABI usage, resources, routes, UI, and typeflow.
  • The React viewer is deliberately thin: DOM and wallet interaction stay outside the protocol packages.
  • A generic integration-fuzz runner walks deployed CAM hosts to catch workflow failures that static checks miss.

Try the current rebuild

The current path is Docker-backed through the Makefile. Dependency installation is explicit and may take time. The GUI target starts a local Anvil chain, deploys the bike NFT fixture, serves CAM resources, and launches the React viewer through a local gateway. The target prints the URL to open in your browser. 

git clone https://github.com/numpde/dapp32.git
cd dapp32
make deps
make package-deps
make cam-conformance-check
make bike-nft-viewer-gui
# open the printed http://127.0.0.1:5173/... URL

Adjacent ideas

The useful comparison is by layer. Most adjacent work solves naming, transport, wallet execution, or distribution. dapp32 focuses on the missing middle: can an Ethereum contract publish a validated, declarative workflow that a generic client can render and execute without trusting arbitrary app JavaScript?

TokenScript / ERC-5169

Useful here: contract-discovered client resources are the right starting point, and token-aware UIs are a proven need.

Not enough: dapp32 wants a small validated workflow grammar, not an open-ended script surface tied mainly to token rendering.

ERC-4804 web3://

Useful here: contract-addressed reads and URI resolution are a natural way to fetch onchain or contract-mediated resources.

Not enough: a URL can locate data; it does not define routes, typed arguments, UI nodes, write preparation, or workflow conformance.

CCIP Read / ERC-3668

Useful here: OffchainLookup is a strong pattern for contract-authorized offchain reads with explicit verification back onchain.

Not enough: it is a read delegation mechanism, not a declarative app contract, screen model, or user-action grammar.

ENS contenthash

Useful here: names can point to content-addressed app resources, which is valuable for publication and review.

Not enough: contenthash answers where bytes live; dapp32 still needs to say what those bytes mean to a viewer, wallet, or agent.

ERC-4337 account abstraction

Useful here: UserOperations, bundlers, and paymaster flows are good execution targets for prepared CAM writes.

Not enough: account abstraction changes how actions are paid for and submitted; it does not tell a client which actions an app exposes or how to validate them.

EIP-5792 wallet call batches

Useful here: wallet capabilities and batched calls are a clean handoff once a CAM viewer has prepared write requests.

Not enough: it starts at the wallet API boundary; dapp32 is concerned with how a contract-published workflow produces those requests safely.

EIP-712 typed data

Useful here: typed signing is the right primitive for reviewable approvals, delegations, attestations, and agent authorizations.

Not enough: it describes data to sign, not the route graph, rendered workflow, or contract-backed state model that leads to a signing request.

ERC-681 transaction requests

Useful here: simple call/payment links are useful for QR codes and narrow one-shot interactions.

Not enough: dapp32 targets multi-step, contract-defined workflows with typed reads, screens, actions, and conformance checks.

EIP-1193 / EIP-6963

Useful here: provider APIs and multi-wallet discovery are ordinary browser integration points for a CAM viewer.

Not enough: they standardize wallet access, not the app workflow or manifest semantics a wallet-adjacent client should interpret.

Solana Actions / Blinks

Useful here: they show demand for portable action surfaces that clients can preview, render, and turn into transactions.

Not enough: they are API/server-oriented; dapp32 asks whether the Ethereum contract itself can anchor the workflow and validation boundary.

Farcaster Mini Apps

Useful here: Mini Apps are a strong distribution and client-container story for user-facing crypto experiences.

Not enough: they intentionally provide an app runtime; dapp32 explores the smaller protocol layer a generic client could interpret before any app-specific runtime is trusted.

ERC-7572 contract metadata

Useful here: contract-level metadata is another precedent for contracts publishing machine-readable resources about themselves.

Not enough: metadata describes a contract; dapp32 needs an executable-but-declarative workflow that generic viewers can validate step by step.

Historical PoC

The original 2023 proof of concept ran on Polygon Mumbai. Mumbai was sunset in April 2024, so the old live URL is no longer a working deployment. The archived branch remains available as historical context, while active development happens on main.
