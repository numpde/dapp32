Contract-defined workflows

Generic viewers for contract-published app flows.

dapp32 explores CAM: a contract-published, hash-pinned manifest format that lets generic clients render dapp views, navigate read routes, and prepare writes without trusting bespoke frontend JavaScript.

Current CAM flow

1
Host contract

Publishes camURI and camHash.

2
main.json

Declares namespaces, ABI resources, routes, and continuations.

3
ui.json

Declares a small portable UI node catalog.

4
Generic viewer

Loads, validates, renders, simulates, and hands writes to a wallet.

Problem

Smart contracts enforce dapp rules, but users and agents still depend on custom websites to know what to display, what to ask for, what a button does, and what should happen after a transaction.

That frontend is usually an unverified offchain interpreter of the contract's workflow. dapp32 asks whether the workflow itself can be published by the contract, validated as data, and consumed by generic wallets, viewers, and agents.

Approach

Contract-anchored manifests

The host contract points to a CAM bundle. The root manifest and secondary ABI/UI resources are integrity checked before use.

Routes before rendering

Read routes call view functions and pass outputs into a UI catalog. Write routes prepare ABI calls for simulation and wallet review.

Small UI model

The UI resource is declarative: text, fields, addresses, status rows, NFT references, includes, and buttons. No arbitrary app JS.

What works today

  • CAM protocol packages parse manifests, UI resources, expressions, and inert values.
  • The viem adapter loads CAM hosts, verifies resources, resolves contract namespaces, reads routes, and prepares writes.
  • @cam/viewer provides a headless session for route navigation, local state, and action dispatch.
  • @cam/conformance validates checked CAM bundles across manifest, ABI, resource, route, UI, and typeflow boundaries.
  • The bike NFT demo deploys locally, serves CAM resources, and runs in a graphical viewer.
  • A generic integration-fuzz runner walks live CAM deployments with optional local fixture writes.

Try the current rebuild

The current path is Docker-backed through the Makefile. The GUI target starts a local Anvil chain, deploys the bike NFT fixture, serves CAM resources, and launches the React viewer through a local gateway. 

git clone https://github.com/numpde/dapp32.git
cd dapp32
make deps
make package-deps
make cam-conformance-check
make bike-nft-viewer-gui

Project pieces

Protocol packages

@cam/protocol, @cam/core, @cam/screen, @cam/evm-viem, @cam/viewer, and @cam/conformance.

Reference app

cam-web is a thin renderer around the headless viewer. The browser owns DOM and wallet interaction; protocol logic stays in reusable packages.

Demo dapp

dapps/bike-nft exercises registration, lookup, metadata, missing/clear, retirement, local deployment, and CAM UI projection.

Funding roadmap

Stabilize CAM v0.1

Tighten the manifest and UI specs, document authoring rules, and keep conformance diagnostics precise enough for builders to fix bundles before deployment.

Improve builder tooling

Turn the current validation and integration-fuzz lanes into a clear publication workflow for teams building contract-defined apps.

Prove generality

Add a second non-bike demo so CAM is not overfit to one registry workflow.

Harden wallet and agent boundaries

Continue refining prepared writes, simulation diagnostics, and the interface generic clients use before asking a wallet or agent to act.

Adjacent ideas

dapp32 is related to TokenScript/ERC-5169, ERC-4804 web3://, CCIP Read, ENS contenthash, ERC-4337, EIP-5792, Solana Actions/Blinks, and Farcaster Mini Apps. The narrower question here is whether Ethereum contracts can publish validated, declarative workflows that generic clients can interpret without arbitrary app JavaScript.

Historical PoC

The original 2023 proof of concept ran on Polygon Mumbai. Mumbai was sunset in April 2024, so the old live URL is no longer a working deployment. The archived branch remains available as historical context, while active development happens on main.
