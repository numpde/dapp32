dapp32

A proof of concept for making a smart contract carry its own app experience.

The live PoC is currently out of commission.

It was first working on May 26, 2023, on Polygon Mumbai (0x13881). Mumbai was later sunset on April 13, 2024, so the historical demo URL no longer represents a live working deployment. The source remains useful as an archived experiment until the contract and viewer are redeployed to a current testnet.

Purpose

A normal decentralized app usually has a smart contract, a custom frontend website, and sometimes a backend. The contract has the real logic, but users still need a website that understands the intended workflow: labels, form fields, buttons, success screens, and payment options.

dapp32 makes the frontend generic. A contract can say: here is my ABI, here is the first screen, here are the fields to show, here is what a button should call, and here is what to show after success or failure. The viewer reads that description and renders the app.

The goal is portability: the app experience follows the contract instead of living only on one hosted website.

How to open one

A dapp32 route has the shape /ui/[network]/[address]/[view]. That gives the viewer enough information to find a chain, connect to the contract, and ask for the first screen.

How the flow works

1 Open a route

The URL names the chain, contract address, and first view function.

2 Read the ABI

The viewer calls abiURI() and loads the contract interface.

3 Fetch UI JSON

A view or pure function returns the screen definition.

4 Render controls

JSON elements become text, inputs, selects, buttons, QR codes, or NFT views.

5 Call the contract

Buttons trigger read-only calls, wallet transactions, or gasless OpenGSN relay calls.

Core pieces

Contract

Provides abiURI(), view functions, transaction functions, and optional success or failure screens.

Viewer

The Next.js app loads the ABI, prepares variables, renders UI JSON, and dispatches contract calls.

Wallet and relay

Users can sign transactions directly, while gasless actions can route through an OpenGSN paymaster.

Current status

This repository is an early proof of concept. Some UI JSON is still served from HTTP files, and the Next.js viewer is not static-export-ready yet. The active path is the viewer under poc/dapp32-nx and the sample Hardhat contract under poc/on-chain. The first end-to-end working PoC appears in the commit history on May 26, 2023, at 1aad7a9. The open niche for dapp32 is a versioned, declarative, contract-discoverable UI schema. A practical next version should reuse existing standards for discovery, storage, wallet execution, and gas sponsorship, while avoiding arbitrary JavaScript by default.

Monetization

The practical business angle is to help teams ship contract-native apps without building a custom frontend or operating gas infrastructure.

Developer tooling SaaS

Paid tools for creating, validating, previewing, and testing dapp32 UI schemas before they are attached to contracts.

Managed gasless infrastructure

Hosted paymasters or relayers with monitoring, spending limits, abuse controls, analytics, and usage-based pricing.

Verified viewer and discovery

A canonical viewer with contract profiles, verified UI manifests, previews, analytics, search, and custom domains.

Related approaches

There are several nearby standards and systems, but no dominant Ethereum standard for a generic renderer of multi-screen, contract-defined app workflows.

TokenScript and ERC-5169

Closest match, ERC Final

ERC-5169 lets a token contract expose scriptURI(), pointing clients to official executable scripts. TokenScript uses this kind of idea to let wallets render token-specific cards, actions, and mini-apps. It is powerful, but broader and more script-driven than dapp32's declarative UI experiment.

ERC-4804 web3://

ERC Final, experimental adoption

ERC-4804 maps human-readable web3:// URLs to read-only EVM calls, so contracts can serve web content such as HTML, SVG, and data. It overlaps with dapp32's contract-carried presentation goal, but it focuses on URL-to-call translation rather than guided app workflows.

CCIP Read / ERC-3668

ERC Final, used in production

CCIP Read lets contracts request offchain data during reads while retaining contract-side validation. It is relevant to dapp32 because UI JSON could be fetched through a standard offchain lookup path instead of arbitrary HTTP URLs.

ENS contenthash and dweb hosting

Practical, gateway-dependent

ENS contenthash plus IPFS, Arweave, or Swarm can decentralize the frontend itself. This solves hosting and naming, but not the deeper question of how a contract describes its own workflow to any compatible viewer.

EIP-5792 and ERC-4337

Modern wallet execution path

EIP-5792 standardizes wallet capabilities, batched calls, status checks, and paymaster-style capability discovery. ERC-4337 is the mainstream account-abstraction and paymaster direction. Together they are a better long-term execution target than a custom transaction path.

Solana Actions/Blinks and Farcaster Mini Apps

Active adjacent ecosystems

These systems show the same product pressure: make onchain actions available where users already are. They are generally API, social, or platform driven rather than contract-native, but they validate the demand for lightweight transaction-oriented app surfaces.
