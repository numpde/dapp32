Blockchain tooling baseline, 2026+

A practical Docker-first EVM stack for building a contract-defined workflow project from scratch: contract-anchored manifests, static human rendering, and agent-readable workflows.

Target: EVM Compiler: pinned in dapps/foundry.toml Contract framework baseline: Foundry Frontend: static Vite + React Trust boundary: contract-published manifest integrity

Recommended Shape

Toolchain

Docker + Make

Make is the public operator interface. Docker keeps Node, Foundry, browsers, and security tools from leaking into the host environment.

Contracts

Solidity + Foundry

Use the compiler and EVM target pinned in dapps/foundry.toml, plus Forge tests, Anvil local chains, Cast inspection, and OpenZeppelin primitives for common patterns.

Security

Slither + Fuzzing

Run static analysis, unit tests, fuzz tests, and invariant checks in Docker-backed lanes.

Viewer

Vite + React + viem

Keep the human viewer static-hostable. Protocol logic should live outside React so agents and wallets can reuse it.

Essential Tools

Foundry

forge, anvil, and cast form the default contract development loop.

OpenZeppelin Contracts

Use established primitives for access control, ownership, ERC interfaces, and common security patterns.

Slither

Baseline static analysis for Solidity code. Run it in the normal check lane once contracts exist.

Foundry Fuzz and Invariants

Property tests catch workflow and state-machine failures that example tests usually miss.

viem + wagmi

Use viem for typed EVM actions and wagmi for React wallet state. Add RainbowKit or Reown AppKit for wallet connection UX.

JSON Schema

The manifest is a public interface. Validate it strictly before rendering or deriving actions.

Smallest Useful Manifest Loop

1 Publish URI

Contract exposes manifestURI().

2 Publish Hash

Contract exposes a hash or uses content-addressed storage.

3 Fetch Bytes

Viewer downloads the manifest as raw bytes.

4 Verify

Core code checks the bytes against the contract-published commitment.

5 Validate

Core code validates the versioned schema.

6 Render

The viewer renders only validated manifest data.

7 Act

Actions produce calldata only if they match the manifest and local policy.

Security Baseline

Pin the environment.

Pin Docker images, package dependencies, compiler version, and EVM target in committed config. Keep contract builds warning-free.

Do not trust manifests by location.

Contract-returned URLs are untrusted. A fetched manifest becomes usable only after integrity verification and schema validation.

Keep contracts authoritative.

Manifests describe workflows. They do not authorize behavior. The contract must enforce business rules onchain.

Prefer declarative actions.

Avoid arbitrary JavaScript in manifests by default. Use bounded components and explicit calldata/action descriptions.

Optional Later

Hardhat can be added if TypeScript-heavy deployment workflows become central. Echidna and Certora are useful for deeper assurance once meaningful invariants exist. ERC-4337 and paymaster tooling belong after the schema, core verifier, and static viewer foundation is working.
