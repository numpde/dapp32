Local Graphical Viewer Stack

Updated 2026-05-28. The GUI viewer should reuse the current local bike fixture discipline: exact read-only mounts, local-only throwaway keys, Docker-managed infrastructure, and protocol logic kept in CAM packages. The main difference from the terminal viewer is that a browser cannot use Docker-internal service names, so the GUI stack needs loopback-only host endpoints.

Goal

Add a graphical host for the same read-only protocol path already proven by the terminal: CamRoot -> CAM -> route call -> screen JSON -> resolved screen.

The GUI should be a consumer of @cam/viewer, @cam/screen, @cam/evm-viem, and @cam/core. It should not own protocol parsing, expression resolution, route execution, or screen semantics.

Boundary

The current terminal stack can stay Docker-internal because the terminal process runs inside Docker. The GUI runs in a host browser. Therefore, browser-facing RPC, CAM resources, and the Vite dev server must bind only to host loopback addresses.

Use 127.0.0.1 ports, never wildcard host publishing, and do not store Docker-only URLs such as http://bike-nft-cam-http:8080/main.json in CamRoot for the graphical scenario.

Current Building Blocks

File Current job GUI implication
compose/bike-nft/local/deploy.yml Starts internal Anvil and deploys the local bike fixture with an obvious throwaway key. Reusable, but the GUI scenario must pass a browser-reachable CAM_URI.
compose/bike-nft/local/http.yml Serves dapps/bike-nft/cam internally for Docker consumers. Reusable with a GUI overlay that publishes it on 127.0.0.1 and applies a narrow CORS policy.
compose/bike-nft/local/viewer-terminal.yml Runs the real-RPC terminal inside Docker, reads Forge broadcast JSON from a named volume. Do not mutate into a GUI file. Add a sibling GUI overlay with a separate service and port policy.

Recommended Services

bike-nft-anvil

Runs the local EVM chain. For the GUI overlay only, publish RPC on host loopback, for example 127.0.0.1:8545.

No repo mount, no private operator key, no Docker socket, dropped capabilities, read-only rootfs, bounded PIDs and memory.

deploy-bike-nft-local

One-shot Foundry deploy job. It should keep the narrowed mounts already used by deploy.yml: foundry.toml, remappings.txt, dependencies/, bike-nft/, and cam/, all read-only.

Continue using the obvious baba... local fixture key. The job must never accept a real private key or arbitrary live RPC URL.

bike-nft-cam-http

Serves checked-in CAM resources to the browser. For the GUI overlay, publish only 127.0.0.1:8090 and make CamRoot.camURI() point at that browser-reachable origin.

Mount only dapps/bike-nft/cam read-only. Add CORS only for the local viewer origin, not *.

cam-web

Runs the graphical viewer app from apps/cam-web. In local dev, expose Vite only on 127.0.0.1:5173. Keep Vite host checks enabled; do not use broad allowedHosts: true.

No private keys, no transaction sending in the first slice, no Docker socket, no protocol logic copied into React.

Runtime Flow

1
CAM HTTP starts. It serves main.json, abi/, and screens/ from dapps/bike-nft/cam.
2
Anvil starts. The GUI overlay publishes only the host-loopback RPC port needed by the browser.
3
The deploy job runs. It deploys CamRoot and app contracts, stores http://127.0.0.1:8090/main.json as the local camURI, and writes Forge broadcast output to a named volume.
4
The GUI loads runtime config. The app should fetch a small JSON config at startup, not bake local contract addresses into a Vite build. The config can be generated from Forge broadcast JSON by a small Docker-side helper.
5
The GUI follows the package path. React renders the resolved screen tree returned by @cam/viewer. It does not call route contracts directly and does not parse CAM or screen documents itself.

Frontend Shape

Use Vite + React

This viewer is a static client-side tool, not a server-rendered product. A small Vite + React app is the right first GUI. Keep it app-local under apps/cam-web with its own lockfile and node_modules/.

Vite environment variables are build-time client inputs. Use them only for non-sensitive defaults. Local chain addresses and deployment state should be fetched from runtime JSON.

Keep React Boring

React should own interaction state, form controls, and rendering. CAM session state comes from @cam/viewer. Screen semantics come from @cam/screen. EVM reads come from @cam/evm-viem.

Wallet connection and transaction sending should wait until screen action documents are stable. The first GUI remains read-only.

Supply Chain

Runtime services must not install dependencies. Solidity dependency acquisition stays in make deps. CAM package dependency acquisition stays in make package-deps. The future apps/cam-web app should get a separate Docker-backed app-local npm lane with the same stage/apply discipline, not a repo-root npm install and not reuse of packages/node_modules.

Project Dependency state Build/test boundary Runtime relation
packages/ packages/package-lock.json and packages/node_modules/ Existing package-* lanes. Builds reusable CAM libraries.
apps/cam-web/ App-local package-lock.json and node_modules/. Future cam-web-* lanes using the same confinement model. Runs the graphical viewer and imports built CAM packages.

First Compose Draft

Add a sibling overlay, not a mutation of the terminal overlay: compose/bike-nft/local/viewer-gui.yml.

It should compose with deploy.yml and http.yml, publish only loopback ports, redirect Forge broadcast output into the same scenario volume, run a runtime-config helper, then run apps/cam-web.

Rejected Shortcuts

  • Do not mount the repository root into GUI runtime services.
  • Do not use Docker service names in browser-facing URIs.
  • Do not use allowedHosts: true for Vite.
  • Do not expose Anvil or CAM HTTP on 0.0.0.0.
  • Do not put mock contract reads in the GUI.
  • Do not add wallet sending before screen actions are specified.

Definition Of Done

  • apps/cam-web exists as an app-local Vite + React project.
  • The GUI stack starts with loopback-only ports for Vite, Anvil, and CAM HTTP.
  • CamRoot.camURI() is browser-reachable in the GUI scenario.
  • The viewer loads runtime config rather than bundled deployment addresses.
  • The GUI renders screens through @cam/viewer and @cam/screen.
  • No GUI runtime service has secrets, repo-root mounts, Docker socket access, or dependency-install authority.
  • Rendered Compose posture tests assert the GUI ports, mounts, and no-secret boundary.
