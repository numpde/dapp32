Local graphical viewer stack

A real local viewer should use the same trust boundaries as production: browser code talks to a browser-reachable RPC endpoint, fetches browser-reachable CAM resources, and renders data returned by real local contracts. Docker supplies the local infrastructure; it should not become the runtime boundary the browser depends on.

Principle

The graphical viewer is an app, not another protocol package. It should consume @cam/viewer, @cam/screen, and the EVM adapter. It should not reimplement CAM parsing, route resolution, ABI loading, or screen semantics.

The local stack should be real-local rather than mock-local: Anvil is the chain, contracts are deployed, CAM files are served over HTTP, and the browser uses viem against the local RPC.

Critical boundary

The viewer JavaScript runs in the host browser. Any URI stored in CamRoot.camURI() must be reachable from that browser.

A Compose-only URI such as http://cam-static:8080/main.json is wrong for this stack. The browser needs a loopback URI such as http://127.0.0.1:8090/bike-nft/cam/main.json.

Recommended services

anvil

Runs the local EVM chain. It publishes a host-loopback RPC port, typically 127.0.0.1:8545, because browser wallets and browser viem clients run outside Docker.

Use the existing Foundry image. No repo mount, no secrets, no Docker socket, no elevated capabilities.

cam-static

Serves checked-in CAM resources and generated local deployment metadata over host-loopback HTTP. It must serve CORS headers for the viewer origin.

Mount dapps/ and generated deployment metadata read-only. Do not mount secrets.

deploy

One-shot Foundry job. It waits for Anvil, deploys the local fixture, stores the browser-reachable CAM URI in CamRoot, and writes a generated deployment artifact with the local CamRoot address.

It should use a known Anvil development key and a hardcoded local RPC service URL. It should never accept a real operator private key or arbitrary RPC URL for this scenario.

viewer-web

Runs the Vite graphical viewer app. It loads runtime config from the static server, creates a viem client for Anvil, loads @cam/viewer, and renders the resolved screen tree.

No private keys, no transaction sending in the first slice, no Docker socket, and no protocol logic duplicated in React.

Supply-chain lanes

Dependency acquisition is not part of the graphical viewer runtime stack. It remains a separate, explicit operation with egress confined to dedicated proxy-backed Compose lanes.

The viewer runtime should use already-materialized Solidity and npm dependencies. It should not run npm install, forge soldeer install, or any dependency update command while serving the app.

Lane Compose file Networked service Output Runtime relation
make deps compose/deps.yml dependency-egress-proxy plus Soldeer staging dapps/dependencies/, lock metadata, dependency checksums Run before Solidity build/deploy lanes; routine Forge lanes verify with deps-verify.
make package-deps compose/package-deps.yml package-egress-proxy plus npm staging Root node_modules/ and, only with ALLOW_UPDATE=1, package-lock.json Run before package build/test/viewer lanes; routine npm lanes stay offline.
make package-graph-check compose/packages.yml None No new dependency material Checks the installed npm graph before package code runs; it is not a per-file integrity verifier.

Keep runtime and acquisition separate

  • The graphical viewer stack should not have external package registry access.
  • The static server and viewer web server should not be allowed to update lockfiles.
  • The deploy job should depend on deps-verify, not run Soldeer itself.
  • The viewer web server should depend on package-build or an equivalent offline package build, not run npm install.

Runtime flow

Host browser

Loads the viewer app and wallet provider.

->

Anvil RPC

Serves real local contract reads.

->

CAM HTTP

Serves CAM, ABI, screen, and deployment JSON.

What calls what

1
Static server starts first. It makes dapps/bike-nft/cam/main.json, abi/, screens/, and generated deployment metadata reachable from the host browser. The server can start before deployment; the deployment JSON appears once the deploy job writes it.
2
Anvil starts with a host-loopback RPC port. The port is intentionally host-visible because browser wallets and the graphical viewer cannot call Docker-internal service names.
3
The deploy job writes the local contract registry. It deploys CamRoot and app contracts to Anvil, seeds local demonstration state, and writes var/deployments/bike-nft/anvil.json.
4
The viewer loads runtime config. It fetches the deployment artifact from the static server instead of baking local addresses into a build-time environment file.
5
The viewer follows the protocol path. It reads CamRoot.camURI() and CamRoot.camHash(), fetches the CAM document, verifies the hash when nonzero, resolves contract addresses and ABIs, calls a route, fetches the returned screen, and renders the resolved screen.

Privileges and mounts

Service Network Writable state Secrets Why
anvil Host-loopback RPC only Container memory/tmpfs None The browser and wallet need a real local RPC endpoint.
cam-static Host-loopback HTTP only None None The browser needs normal HTTP resources and CORS.
deploy Compose network to Anvil var/deployments/ or a narrow generated artifact mount None; known Anvil development key only The viewer needs the local CamRoot address after deployment.
viewer-web Host-loopback dev server None for runtime; package build output only in normal package lanes None The browser app should contain no private material.

Runtime config

Prefer a generated deployment artifact over build-time Vite constants. The viewer can fetch:

http://127.0.0.1:8090/deployments/bike-nft/anvil.json

That artifact should contain the chain id, RPC URL, local CamRoot address, and browser-reachable CAM URI.

Keep it runtime-loaded, not bundled into the app: {"chainId":31337,"rpcUrl":"http://127.0.0.1:8545","camRoot":"0x...","camURI":"http://127.0.0.1:8090/bike-nft/cam/main.json"}

Rejected shortcuts

  • Do not use Compose-only service names in on-chain CAM URIs.
  • Do not put mock contract reads in the graphical viewer.
  • Do not give the viewer or static server private keys.
  • Do not mount the Docker socket.
  • Do not duplicate CAM or screen protocol logic in React.

First Make target shape

The first local graphical target should be separate from the mock terminal:

make bike-nft-viewer-local

It should depend on the package build and dependency integrity gates, including package-build for TypeScript packages and deps-verify before any Foundry compile/deploy work. Then it should start Anvil and the static server, run the local deploy job, and start the viewer web server. A matching make bike-nft-viewer-local-down should stop the stack and remove only this scenario's containers and generated volumes.

Definition of done

  • The browser loads the viewer from a loopback dev server.
  • The browser reads real local contracts through Anvil.
  • The CAM URI stored in CamRoot is browser-reachable.
  • CAM, ABI, screen, and deployment JSON are served by a local static service.
  • The viewer renders route screens through @cam/viewer and @cam/screen.
  • The first slice remains read-only: no wallet transaction sending yet.
