Local Graphical Viewer Stack

Updated 2026-06-09. The local GUI viewer is an implemented bike NFT scenario: Docker starts Anvil, deploys the fixture, serves checked-in CAM resources, runs the React viewer, and exposes exactly one browser gateway. Protocol logic stays in CAM packages; the browser is only a renderer and wallet boundary.

Goal

Document the graphical host for the protocol path first proven by the terminal: CamRoot -> CAM -> route call -> UI catalog -> resolved UI tree.

The GUI is a consumer of @cam/viewer, @cam/screen, @cam/evm-viem, and @cam/core. It does not own protocol parsing, expression resolution, route execution, or UI semantics.

Boundary

The current terminal stack can stay Docker-internal because the terminal process runs inside Docker. The GUI runs in a host browser. Therefore, browser-facing RPC, CAM resources, and the Vite dev server are reached through one configured browser origin.

Default to 127.0.0.1 port publishing. If the browser runs elsewhere, set the bind host and browser origin deliberately. Do not store Docker-only URLs such as http://bike-nft-cam-http:8080/main.json in CamRoot for the graphical scenario.

Running Services

bike-nft-anvil

Runs the local EVM chain. In the GUI stack it stays internal; the browser reaches it through gateway path /rpc.

No repo mount, no private operator key, no Docker socket, dropped capabilities, read-only rootfs, bounded PIDs and memory.

deploy-bike-nft-local

One-shot Foundry deploy job. It keeps the narrowed mounts in deploy.yml: foundry.toml, remappings.txt, dependencies/, bike-nft/, and cam/, all read-only.

Continue using the obvious baba... local fixture key. The job must never accept a real private key or arbitrary live RPC URL.

bike-nft-cam-http

Serves checked-in CAM resources internally. In the GUI stack, the browser reaches those files through gateway path /cam/, and CamRoot.camURI() points at that browser-reachable route.

Mount only dapps/bike-nft/cam read-only. No CORS policy is needed in this scenario because the browser uses a same-origin gateway.

cam-web

Runs the graphical viewer app from js/apps/cam-web. Vite stays Docker-internal and only the gateway is exposed. Keep Vite host checks enabled; do not use broad allowedHosts: true.

No private keys, no Docker socket, no protocol logic copied into React. Transaction sending must go through an injected browser wallet.

bike-nft-browser-gateway

Publishes the only browser-facing host port, defaulting to 127.0.0.1:5173. Routes / to Vite, /rpc to Anvil, and /cam/ to the CAM resource server.

This keeps the real services on the internal fixture network and gives the browser a same-origin local app.

Runtime Flow

1
CAM HTTP starts. It serves main.json, ui.json, and abi/ from dapps/bike-nft/cam.
2
Anvil starts. It stays on the internal Docker network. The browser reaches JSON-RPC through gateway path /rpc.
3
The deploy job runs. It deploys CamRoot and app contracts, stores BIKE_NFT_GUI_ORIGIN/cam/main.json as the local camURI, and writes Forge broadcast output to a named volume.
4
The GUI reads URL startup parameters. The browser URL supplies the chain id, RPC URL, CamRoot address, and account. Unsigned CAM acceptance is a build/runtime policy, not a URL parameter. The app then reads camURI, camHash, and contract addresses from CamRoot.
5
The GUI follows the package path. React renders the resolved UI tree returned by @cam/viewer. It does not call route contracts directly and does not parse CAM or UI documents itself.

Frontend Shape

Use Vite + React

This viewer is a static client-side tool, not a server-rendered product. The local GUI is a small Vite + React app inside the shared js/ workspace.

Vite environment variables are scenario policy, not secrets. The GUI uses them for unsigned-CAM policy and the pinned resource origin. Local chain addresses and deployment state come from the browser URL or from the chain, not from a baked Vite build.

Keep React Boring

React owns rendering and DOM events. CAM session state comes from @cam/viewer. UI element semantics come from @cam/screen: TextField.state wires local state, and Button.call dispatches through CAM routes. EVM reads come from @cam/evm-viem.

Wallet connection belongs at the app edge. React asks @cam/viewer for a prepared contract call, simulates it, and sends only through the injected browser wallet.

Supply Chain

Runtime services must not install dependencies. Solidity dependency acquisition stays in make deps. CAM package dependency acquisition stays in make package-deps. The js/apps/cam-web app is part of that same JS workspace, so it must use the shared lock and not add a nested app lock.

How To Run

Start the local graphical scenario with: make bike-nft-viewer-gui.

The target composes deploy.yml, http.yml, and viewer-gui.yml. It starts Anvil and CAM HTTP, deploys the fixture, prints the browser URL, then runs cam-web and the gateway in the foreground.

Stop and remove the scenario with make bike-nft-viewer-gui-down. If the browser is not on the same host, set both BIKE_NFT_GUI_BIND_HOST and BIKE_NFT_GUI_ORIGIN deliberately.

Rejected Shortcuts

  • Do not mount the repository root into GUI runtime services.
  • Do not use Docker service names in browser-facing URIs.
  • Do not use allowedHosts: true for Vite.
  • Do not expose Anvil or CAM HTTP on 0.0.0.0.
  • Do not put mock contract reads in the GUI.
  • Do not send transactions from Docker services or CAM packages; use the injected browser wallet boundary.

Operational Checks

  • make package-test builds and tests CAM packages plus cam-web.
  • make checks asserts repository and Compose posture, including GUI stack mounts and ports.
  • make bike-nft-viewer-gui prints a URL with chainId, host, account, and rpcUrl.
  • The browser loads /cam/main.json and /rpc through the same configured origin.
  • Writes are prepared and simulated by the app, then sent only through an injected wallet.
