Local Graphical Viewer Stack

Updated 2026-05-28. The GUI viewer should reuse the current local bike fixture discipline: exact read-only mounts, local-only throwaway keys, Docker-managed infrastructure, and protocol logic kept in CAM packages. The main difference from the terminal viewer is that a browser cannot use Docker-internal service names, so the GUI stack needs one loopback-only browser gateway.

Goal

Add a graphical host for the protocol path first proven by the terminal: CamRoot -> CAM -> route call -> screen JSON -> resolved screen.

The GUI should be a consumer of @cam/viewer, @cam/screen, @cam/evm-viem, and @cam/core. It should not own protocol parsing, expression resolution, route execution, or screen semantics.

Boundary

The current terminal stack can stay Docker-internal because the terminal process runs inside Docker. The GUI runs in a host browser. Therefore, browser-facing RPC, CAM resources, and the Vite dev server should be reached through one host-loopback gateway.

Use 127.0.0.1 ports, never wildcard host publishing, and do not store Docker-only URLs such as http://bike-nft-cam-http:8080/main.json in CamRoot for the graphical scenario.

Recommended Services

bike-nft-anvil

Runs the local EVM chain. In the GUI overlay it stays internal; the browser reaches it through gateway path /rpc.

No repo mount, no private operator key, no Docker socket, dropped capabilities, read-only rootfs, bounded PIDs and memory.

deploy-bike-nft-local

One-shot Foundry deploy job. It should keep the narrowed mounts already used by deploy.yml: foundry.toml, remappings.txt, dependencies/, bike-nft/, and cam/, all read-only.

Continue using the obvious baba... local fixture key. The job must never accept a real private key or arbitrary live RPC URL.

bike-nft-cam-http

Serves checked-in CAM resources internally. In the GUI overlay, the browser reaches those files through gateway path /cam/, and CamRoot.camURI() points at that browser-reachable route.

Mount only dapps/bike-nft/cam read-only. No CORS policy is needed in this scenario because the browser uses a same-origin gateway.

cam-web

Runs the graphical viewer app from js/apps/cam-web. In local dev, keep Vite Docker-internal and expose the gateway on 127.0.0.1:5173. Keep Vite host checks enabled; do not use broad allowedHosts: true.

No private keys, no Docker socket, no protocol logic copied into React. Transaction sending must go through an injected browser wallet.

bike-nft-browser-gateway

Publishes the only browser-facing host port: 127.0.0.1:5173. Routes / to Vite, /rpc to Anvil, and /cam/ to the CAM resource server.

This keeps the real services on the internal fixture network and gives the browser a same-origin local app.

Runtime Flow

1
CAM HTTP starts. It serves main.json, abi/, and screens/ from dapps/bike-nft/cam.
2
Anvil starts. It stays on the internal Docker network. The browser reaches JSON-RPC through gateway path /rpc.
3
The deploy job runs. It deploys CamRoot and app contracts, stores http://127.0.0.1:5173/cam/main.json as the local camURI, and writes Forge broadcast output to a named volume.
4
The GUI reads URL startup parameters. The browser URL supplies the chain id, RPC URL, CamRoot address, account, and unsigned-hash policy. The app then reads camURI, camHash, and contract addresses from CamRoot.
5
The GUI follows the package path. React renders the resolved screen tree returned by @cam/viewer. It does not call route contracts directly and does not parse CAM or screen documents itself.

Frontend Shape

Use Vite + React

This viewer is a static client-side tool, not a server-rendered product. A small Vite + React app is the right first GUI. Keep it inside the shared js/ workspace.

Vite environment variables are build-time client inputs. Use them only for non-sensitive defaults. Local chain addresses and deployment state should come from the browser URL or from the chain, not from a baked Vite build.

Keep React Boring

React should own interaction state, form controls, and rendering. CAM session state comes from @cam/viewer. Screen semantics come from @cam/screen. EVM reads come from @cam/evm-viem.

Wallet connection belongs at the app edge. React can ask @cam/viewer for a prepared contract call, but the injected wallet remains the only sender.

Supply Chain

Runtime services must not install dependencies. Solidity dependency acquisition stays in make deps. CAM package dependency acquisition stays in make package-deps. The js/apps/cam-web app is part of that same JS workspace, so it must use the shared lock and not add a nested app lock.

First Compose Draft

Add a sibling overlay, not a mutation of the terminal overlay: compose/bike-nft/local/viewer-gui.yml.

It should compose with deploy.yml and http.yml, publish one loopback gateway port, make CamRoot.camURI() browser-reachable, then run js/apps/cam-web through the staged JS workspace.

Rejected Shortcuts

  • Do not mount the repository root into GUI runtime services.
  • Do not use Docker service names in browser-facing URIs.
  • Do not use allowedHosts: true for Vite.
  • Do not expose Anvil or CAM HTTP on 0.0.0.0.
  • Do not put mock contract reads in the GUI.
  • Do not send transactions from Docker services or CAM packages; use the injected browser wallet boundary.

Definition Of Done

  • js/apps/cam-web exists as a Vite + React workspace app.
  • The GUI stack starts with one loopback-only gateway port.
  • CamRoot.camURI() is browser-reachable in the GUI scenario.
  • The viewer reads URL startup parameters rather than bundled deployment addresses.
  • The GUI renders screens through @cam/viewer and @cam/screen.
  • No GUI runtime service has secrets, repo-root mounts, Docker socket access, or dependency-install authority.
  • Rendered Compose posture tests assert the GUI ports, mounts, and no-secret boundary.
