How the dependency lane works

make deps is the controlled dependency lane. By default it installs only the currently locked dependency set, applies declared local patches, and fails if lock, remapping, or checksum metadata would change.

Current source: Soldeer Network: stage only Explicit archive check: soldeer.lock Patch check: pre / patch / post SHA-256 Updates: ALLOW_UPDATE=1 Build checks: offline verify first

The rule

Dependency installation is split into stages because downloading code, changing installed dependency bytes, and writing repository files have different trust boundaries. The networked stage downloads dependencies, verifies each upstream archive against soldeer.lock, applies declared local patches, writes checksums for the final patched tree, and writes only to a temporary Docker volume. Offline apply and verify containers handle repository writes and checks.

The important invariant is simple: no networked dependency process gets repo-root read/write access.

What runs where

The public entrypoint is make. Make invokes Compose services; the services run the actual tools inside constrained containers.

1. Egress

dependency-egress-proxy

Image: containers/https-egress-proxy. This is the only service with external network egress. It allows HTTPS CONNECT traffic to public port 443.

2. Stage

soldeer-stage

Image: containers/foundry-deps. Runs forge soldeer install, explicitly verifies the upstream ZIP against soldeer.lock, applies declared patches, then writes patched checksums. Writes only to soldeer-out.

3. Apply

soldeer-apply-locked

Image: containers/foundry. No network. Copies patched dependency output from soldeer-out into repo dependencies/. Fails if the staged soldeer.lock, remappings, or checksums differ from the committed files.

4. Update

soldeer-apply-update

Image: containers/foundry. No network. Used only by make deps ALLOW_UPDATE=1. This is the only dependency-lane service allowed to copy a staged soldeer.lock back into the repo.

5. Verify

soldeer-verify

Image: containers/checks. No network, read-only mounts. Runs test_dependency_integrity.py /work against the final installed dependency tree.

6. Build/Test

forge-build / forge-test

Image: containers/foundry. No network, repo read-only. Runs normal forge build --offline or forge test --offline after the dependency verifier has accepted the installed tree.

make deps
  dependency-egress-proxy
  soldeer-stage
    forge soldeer install --config-location foundry
      Soldeer installs from its package metadata and lockfile.
    test_dependency_integrity.py --verify-upstream /tmp/soldeer-work
      This is the explicit local-hash check for the downloaded ZIP.
    apply_dependency_patches
    test_dependency_integrity.py --write-checksums /tmp/soldeer-work
  soldeer-apply-locked OR soldeer-apply-update
  soldeer-verify

make build
  soldeer-verify
  forge build --offline

make test
  soldeer-verify
  checks
  forge test --offline --match-path contracts/test/unit/**/*.sol -vvv

What gets updated

In locked mode, the Soldeer lane may rewrite only dependencies/ to match the committed metadata. With ALLOW_UPDATE=1, it may update exactly these repository outputs:

dependencies/

The ignored installed dependency tree used by Foundry.

soldeer.lock

The Soldeer lockfile. It records package names, versions, source URLs, and upstream package checksums.

remappings.txt

The import remappings generated for Foundry.

dependency-checksums.txt

The committed installed-tree checksums for the final dependency tree that Foundry reads. If declared patches exist, this checksum describes the patched tree.

dependency-patches.txt and patches/

Reviewable patch metadata and patch files. They are committed source inputs, not generated outputs. Update them deliberately, then run make deps ALLOW_UPDATE=1 to refresh the final dependency checksum.

When soldeer.lock changes

forge soldeer install runs in the networked staging container against a temporary working tree. It may produce or refresh a staged soldeer.lock there, but that file lives in the Docker volume until the offline apply step decides what to do with it.

Plain make deps is locked mode. It compares the staged soldeer.lock with the committed soldeer.lock and fails if they differ. It does not update the repository lockfile.

make deps ALLOW_UPDATE=1 is update mode. After staging, the offline soldeer-apply-update service may copy the staged soldeer.lock, remappings.txt, and dependency-checksums.txt back into the repo. That is the deliberate path for adding, removing, or changing Soldeer dependencies.

Declared patches

Soldeer has package-level integrity, not a first-class local patch model. For short-lived local fixes, this repo keeps upstream package provenance in soldeer.lock and records the local change separately in dependency-patches.txt. 

<dependency-key>  <target-path>  <patch-path>  <pre-sha256>  <patch-sha256>  <post-sha256>
pre-sha256

The expected hash of the pristine file installed from the verified Soldeer package.

patch-sha256

The expected hash of the committed patch file.

post-sha256

The expected hash of the installed file after the patch is applied.

Patches are applied with zero fuzz. If upstream changes enough that a patch no longer applies exactly, dependency installation fails and the patch must be reviewed.

What verification proves

During staging, tests/checks/test_dependency_integrity.py reads soldeer.lock, downloads each package URL over HTTPS, checks the archive SHA-256 against the lockfile checksum, extracts the archive, and confirms the installed dependency tree matches the verified archive payload before patches are applied.

After declared patches are applied, the verifier writes dependency-checksums.txt from the final tree. In offline mode, it checks that dependencies/ contains exactly the expected dependency directories. It rejects missing dependency directories, unexpected dependency directories, stray top-level files, duplicate lock entries, missing remappings, stale checksum entries, checksum mismatches, patch-file hash drift, and patched-target hash drift.

The local checksum for each dependency is a deterministic SHA-256 over a sorted manifest of every file path and file hash inside that dependency directory. This is intentionally independent of Soldeer internals. It verifies the installed tree that Foundry will actually read.

Where hash checks happen

Soldeer records local package hashes in soldeer.lock. The lock entry includes the package URL, a SHA-256 checksum for the downloaded ZIP, and an integrity hash for the unpacked dependency folder. Soldeer uses that package metadata for installation and can later warn when the installed package no longer matches its pristine package identity.

The explicit first-download check in this lane is test_dependency_integrity.py --verify-upstream, which runs immediately after forge soldeer install and before any patches. It reads the committed soldeer.lock, downloads the locked HTTPS URL, checks the downloaded ZIP against the lockfile checksum, extracts it, and confirms the installed dependency tree matches that verified archive payload.

After a patch is applied, normal Forge commands may still print Soldeer’s generic FailedIntegrity warning because the installed package no longer matches the pristine package. That warning is expected for a declared patch. The repository verifier is the patch-aware check: it verifies the pristine target hash, the patch-file hash, the patched target hash, and the final dependency-tree checksum before build or test lanes run.

Routine commands

# Install the locked Soldeer set, apply declared patches, and verify the result.
make deps

# Explicitly allow soldeer.lock, remapping, and checksum updates.
make deps ALLOW_UPDATE=1

# Check the installed patched dependency tree without network.
make deps-verify

# Compile or test. These run deps-verify first.
make build
make test
make ci

Reading warnings

A patched Soldeer dependency can trigger Forge’s package-level warning during normal compile or test commands: 

Warning: Dependency '@openzeppelin-contracts' integrity check failed: FailedIntegrity

That warning is generic. It means the installed package no longer matches Soldeer’s pristine package identity. It does not say whether the difference is intended. The repository verifier prints the precise patch explanation before Forge runs: 

deps-verify: @openzeppelin-contracts-5.6.1 declared patch ok
deps-verify:   target: utils/cryptography/ECDSA.sol
deps-verify:   patch: patches/dependencies/@openzeppelin-contracts-5.6.1/ECDSA.error-keyword.patch
deps-verify:   Forge may also warn FailedIntegrity for this declared patch.

Adding more Soldeer dependencies

For another Soldeer dependency, add it through the normal Soldeer configuration path, then run make deps ALLOW_UPDATE=1. The command should regenerate soldeer.lock, remappings.txt, dependencies/, and dependency-checksums.txt. Review all committed outputs before committing. 

make deps ALLOW_UPDATE=1
make ci
git diff -- foundry.toml soldeer.lock remappings.txt dependency-checksums.txt

Adding or removing a patch

Prefer an upstream release or a pinned fork for long-lived dependency changes. Use local patches only for narrow, temporary fixes where the changed file and hashes can be reviewed directly.

Add the patch file.

Place a unified diff under patches/dependencies/<dependency-key>/. Keep it small and targeted.

Update dependency-patches.txt.

Record the dependency key, target path, patch path, pristine target hash, patch-file hash, and patched target hash.

Refresh the final checksum.

Run make deps ALLOW_UPDATE=1, then review dependency-checksums.txt and the patch files before committing.

Adding a different dependency source

make deps is allowed to be the umbrella command, but each dependency system should have its own contained lane. Do not put npm, git submodules, code generation, ABI downloads, or other package managers inside the Soldeer lane.

Add a separate stage service.

The networked service reads only its input files and writes only to its own Docker volume.

Add a separate apply service.

The offline apply service writes only the exact repository outputs owned by that dependency source.

Add a separate verify service.

The offline verifier checks the generated state before any build or test consumes it.

Wire it into make deps and make deps-verify.

The public interface can stay simple while the trust boundaries remain explicit. Default installation must stay locked; metadata updates need an explicit opt-in such as ALLOW_UPDATE=1.

What this does not prove

This lane proves that the fetched archive matches the committed lockfile checksum, that declared patches match their committed hashes, and that the final installed dependency tree matches dependency-checksums.txt. It does not prove that an upstream package or local patch is correct, audited, or safe to use. Reviewing dependency changes is still a human responsibility.
