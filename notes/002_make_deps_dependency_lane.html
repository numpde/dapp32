How the dependency lane works

make deps is the controlled dependency lane. By default it installs only the currently locked dependency set, applies declared local patches, and fails if lock, remapping, or checksum metadata would change.

Current source: Soldeer Network: allowlisted proxy Explicit archive check: soldeer.lock Source policy: foundry.toml plus soldeer.lock Patch check: pre / patch / post SHA-256 Updates: ALLOW_UPDATE=1 Live proof: check-live-deps-egress

The rule

Dependency installation is split into stages because downloading code, changing installed dependency bytes, and writing repository files have different trust boundaries. The networked stage reaches the internet only through a dependency egress proxy, verifies dependency source policy from dapps/foundry.toml and dapps/soldeer.lock, verifies each upstream archive against dapps/soldeer.lock, applies declared local patches, writes checksums for the final patched tree, and writes only to a temporary Docker volume. Offline apply and verify containers handle repository writes and checks.

The important invariant is simple: no networked dependency process gets repo-root read/write access.

What runs where

The public entrypoint is make. Make invokes Compose services; the services run the actual tools inside constrained containers.

1. Egress

dependency-egress-proxy

Image: containers/https-egress-proxy. This is the only service with external network egress. It allows HTTPS CONNECT only to configured global hosts on port 443. Today the dependency lane allowlist is soldeer-revisions.s3.amazonaws.com.

2. Stage

soldeer-stage

Image: containers/foundry-deps. Runs forge soldeer install through the proxy, verifies the configured source policy and upstream ZIP against dapps/foundry.toml and dapps/soldeer.lock, applies declared patches, then writes patched checksums. Writes only to soldeer-out.

3. Apply

soldeer-apply-locked

Image: containers/foundry. No network. Stages patched dependency output under dapps/dependencies/.next, then replaces the visible dependency children. Fails first if the staged dapps/soldeer.lock, remappings, or checksums differ from the committed files.

4. Update

soldeer-apply-update

Image: containers/foundry. No network. Used only by make deps ALLOW_UPDATE=1. It uses the same staged dependency-tree replacement and is the only dependency-lane service allowed to copy staged metadata back into the repo.

5. Verify

soldeer-verify

Image: containers/checks. No network, read-only mounts. Runs test_dependency_integrity.py /work against dapps/foundry.toml, dapps/soldeer.lock, and the final installed dependency tree.

6. Build/Test

forge-build / forge-test

Image: containers/foundry. No network, repo read-only. Runs forge build --offline over every first-level dapps/*/src directory, or forge test --offline over the shared */test/unit/**/*.sol convention, after the dependency verifier has accepted the installed tree.

make deps
  dependency-egress-proxy
  soldeer-stage
    forge soldeer install --config-location foundry
      Soldeer installs from its package metadata and lockfile.
    test_dependency_integrity.py --verify-upstream /tmp/soldeer-work
      Check foundry.toml and soldeer.lock source policy.
      Download each locked ZIP and check its SHA-256.
    apply_dependency_patches
    test_dependency_integrity.py --write-checksums /tmp/soldeer-work
  soldeer-apply-locked OR soldeer-apply-update
    stage_dependency_tree under dependencies/.next
  soldeer-verify

make build
  soldeer-verify
  forge build --offline */src

make test
  soldeer-verify
  checks
  forge test --offline --match-path '*/test/unit/**/*.sol' -vvv

What gets updated

In locked mode, the Soldeer lane may rewrite only dapps/dependencies/ to match the committed metadata. With ALLOW_UPDATE=1, it may update exactly these repository outputs:

dapps/dependencies/

The ignored installed dependency tree used by Foundry.

dapps/soldeer.lock

The Soldeer lockfile. It records package names, versions, source URLs, and upstream package checksums.

dapps/remappings.txt

The import remappings generated for Foundry.

dapps/dependency-checksums.txt

The committed installed-tree checksums for the final dependency tree that Foundry reads. If declared patches exist, this checksum describes the patched tree.

dapps/dependency-patches.txt and dapps/_patches/

Reviewable patch metadata and patch files. They are committed source inputs, not generated outputs. Update them deliberately, then run make deps ALLOW_UPDATE=1 to refresh the final dependency checksum.

When soldeer.lock changes

forge soldeer install runs in the networked staging container against a temporary working tree. It may produce or refresh a staged dapps/soldeer.lock there, but that file lives in the Docker volume until the offline apply step decides what to do with it.

Plain make deps is locked mode. It compares the staged dapps/soldeer.lock with the committed dapps/soldeer.lock and fails if they differ. It does not update the repository lockfile.

make deps ALLOW_UPDATE=1 is update mode. After staging, the offline soldeer-apply-update service may copy the staged dapps/soldeer.lock, dapps/remappings.txt, and dapps/dependency-checksums.txt back into the repo. That is the deliberate path for adding, removing, or changing Soldeer dependencies.

Both apply modes replace the installed dependency tree through a temporary dapps/dependencies/.next directory. That keeps the generated tree separate from the visible tree until the metadata checks or update-mode decision have passed.

Declared patches

Soldeer has package-level integrity, not a first-class local patch model. For short-lived local fixes, this repo keeps upstream package provenance in dapps/soldeer.lock and records the local change separately in dapps/dependency-patches.txt. 

<dependency-key>  <target-path>  <patch-path>  <pre-sha256>  <patch-sha256>  <post-sha256>
pre-sha256

The expected hash of the pristine file installed from the verified Soldeer package.

patch-sha256

The expected hash of the committed patch file.

post-sha256

The expected hash of the installed file after the patch is applied.

Patches are applied with zero fuzz. If upstream changes enough that a patch no longer applies exactly, dependency installation fails and the patch must be reviewed.

What verification proves

During staging, tests/checks/test_dependency_integrity.py reads dapps/foundry.toml and dapps/soldeer.lock. It rejects dependency entries that are not version-only in dapps/foundry.toml, rejects name or version drift between dapps/foundry.toml and dapps/soldeer.lock, and rejects locked URLs outside the approved Soldeer revisions archive shape. It then downloads each package URL over HTTPS, checks the archive SHA-256 against the lockfile checksum, extracts the archive, and confirms the installed dependency tree matches the verified archive payload before patches are applied.

After declared patches are applied, the verifier writes dapps/dependency-checksums.txt from the final tree. In offline mode, it checks that dapps/dependencies/ contains exactly the expected dependency directories. It rejects missing dependency directories, unexpected dependency directories, stray top-level files, duplicate lock entries, missing remappings, stale checksum entries, checksum mismatches, patch-file hash drift, patched-target hash drift, disallowed OpenZeppelin import paths, and source-repo-shaped OpenZeppelin payloads.

The local checksum for each dependency is a deterministic SHA-256 over a sorted manifest of every file path and file hash inside that dependency directory. This is intentionally independent of Soldeer internals. It verifies the installed tree that Foundry will actually read.

Where hash checks happen

Soldeer records local package hashes in dapps/soldeer.lock. The lock entry includes the package URL, a SHA-256 checksum for the downloaded ZIP, and an integrity hash for the unpacked dependency folder. Soldeer uses that package metadata for installation and can later warn when the installed package no longer matches its pristine package identity.

The explicit first-download check in this lane is test_dependency_integrity.py --verify-upstream, which runs immediately after forge soldeer install and before any patches. It first checks the committed dependency source policy, then downloads the locked HTTPS URL, checks the downloaded ZIP against the lockfile checksum, extracts it, and confirms the installed dependency tree matches that verified archive payload.

After a patch is applied, normal Forge commands may still print Soldeer’s generic FailedIntegrity warning because the installed package no longer matches the pristine package. That warning is expected for a declared patch. The repository verifier is the patch-aware check: it verifies the pristine target hash, the patch-file hash, the patched target hash, and the final dependency-tree checksum before build or test lanes run.

Check lanes

make checks

Runs offline repository/source checks in the checks container. This is the default static-policy lane and does not intentionally reach the external network.

make check-runtime

Runs local Docker-backed runtime checks that need a renderer or runtime but should still avoid external network. Today this includes the rendered Anvil Compose posture checks.

make check-live

Umbrella for live checks that intentionally use external network. Keep this separate from default CI unless the runner is expected to have network access.

make check-live-deps-egress

Starts the real dependency-egress-proxy from compose/deps.yml plus the live-check overlay. The checker has no direct egress, derives allowed dependency hosts from dapps/soldeer.lock, proves CONNECT to those hosts succeeds, and proves CONNECT to example.com is denied.

Routine commands

# Install the locked Soldeer set, apply declared patches, and verify the result.
make deps

# Explicitly allow soldeer.lock, remapping, and checksum updates.
make deps ALLOW_UPDATE=1

# Check the installed patched dependency tree without network.
make deps-verify

# Run offline repository/source checks.
make checks

# Run local Docker-backed runtime checks that do not intentionally hit the internet.
make check-runtime

# Prove live dependency-egress allow/deny behavior through the real proxy.
make check-live-deps-egress

# Run all live external-network checks.
make check-live

# Compile or test. These run deps-verify first.
make build
make test
make ci

Reading warnings

A patched Soldeer dependency can trigger Forge’s package-level warning during normal compile or test commands: 

Warning: Dependency '@openzeppelin-contracts' integrity check failed: FailedIntegrity

That warning is generic. It means the installed package no longer matches Soldeer’s pristine package identity. It does not say whether the difference is intended. The repository verifier prints the precise patch explanation before Forge runs: 

deps-verify: @openzeppelin-contracts-5.6.1 declared patch ok
deps-verify:   target: utils/cryptography/ECDSA.sol
deps-verify:   patch: dapps/_patches/dependencies/@openzeppelin-contracts-5.6.1/ECDSA.error-keyword.patch
deps-verify:   Forge may also warn FailedIntegrity for this declared patch.

Adding more Soldeer dependencies

For another Soldeer dependency, add it through the normal Soldeer configuration path, then run make deps ALLOW_UPDATE=1. The command should regenerate dapps/soldeer.lock, dapps/remappings.txt, dapps/dependencies/, and dapps/dependency-checksums.txt. Review all committed outputs before committing.

The source policy is intentionally narrow. A normal Soldeer dependency should lock to soldeer-revisions.s3.amazonaws.com. If a dependency introduces a different locked URL host or archive shape, the verifier and proxy allowlist should fail until that new source is reviewed and deliberately added. 

make deps ALLOW_UPDATE=1
make ci
make check-live-deps-egress
git diff -- dapps/foundry.toml dapps/soldeer.lock dapps/remappings.txt dapps/dependency-checksums.txt

Adding or removing a patch

Prefer an upstream release or a pinned fork for long-lived dependency changes. Use local patches only for narrow, temporary fixes where the changed file and hashes can be reviewed directly.

Add the patch file.

Place a unified diff under dapps/_patches/dependencies/<dependency-key>/. Keep it small and targeted.

Update dapps/dependency-patches.txt.

Record the dependency key, target path, patch path, pristine target hash, patch-file hash, and patched target hash.

Refresh the final checksum.

Run make deps ALLOW_UPDATE=1, then review dapps/dependency-checksums.txt and the patch files before committing.

Adding a different dependency source

make deps is allowed to be the umbrella command, but each dependency system should have its own contained lane. Do not put npm, git submodules, code generation, ABI downloads, or other package managers inside the Soldeer lane.

Add a separate stage service.

The networked service reads only its input files and writes only to its own Docker volume.

Add a separate apply service.

The offline apply service writes only the exact repository outputs owned by that dependency source.

Add a separate verify service.

The offline verifier checks the generated state before any build or test consumes it.

Add a separate egress policy.

If the source needs network access, keep its host allowlist specific and add a live allow/deny proof for that proxy behavior.

Wire it into make deps and make deps-verify.

The public interface can stay simple while the trust boundaries remain explicit. Default installation must stay locked; metadata updates need an explicit opt-in such as ALLOW_UPDATE=1.

What this does not prove

This lane proves that the fetched archive matches the committed lockfile checksum, that declared patches match their committed hashes, and that the final installed dependency tree matches dapps/dependency-checksums.txt. The live egress check proves the configured proxy can reach the current locked dependency hosts and denies one non-allowlisted host. It does not prove that an upstream package or local patch is correct, audited, or safe to use, and it does not prove that future hosts will remain reachable. Reviewing dependency changes is still a human responsibility.
