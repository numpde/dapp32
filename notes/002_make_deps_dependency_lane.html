How make deps works

make deps is the controlled dependency lane. By default it installs only the currently locked dependency set and fails if lock, remapping, or checksum metadata would change.

Current source: Soldeer Network: stage only Upstream check: soldeer.lock Updates: ALLOW_UPDATE=1 Build checks: offline verify first

The rule

Dependency installation is split into stages because the risky part and the persistent-write part should not happen in the same container. The networked container downloads dependencies, verifies each upstream archive against soldeer.lock, writes the local checksum manifest, and writes only to a temporary Docker volume. The offline apply container either accepts the staged metadata as matching the committed metadata, or fails unless ALLOW_UPDATE=1 was set. A separate offline verifier checks the result before normal compile and test lanes run.

The important invariant is simple: no networked dependency process gets repo-root read/write access.

The current flow

1. Stage

soldeer-stage

Runs forge soldeer install with network access. It can read only dependency input files, verifies downloaded archives against soldeer.lock, and writes generated output into a Docker volume.

2. Apply

soldeer-apply-locked

Runs with no network by default. It can write dependencies/, but mounts lock, remapping, and checksum metadata read-only and fails if they would change.

3. Update

soldeer-apply-update

Runs only when make deps ALLOW_UPDATE=1 is used. It can copy staged lock, remapping, and checksum updates into the repository.

4. Verify

soldeer-verify

Runs with no network and read-only mounts. It checks installed dependency directories, remappings, and committed file-manifest checksums.

What gets updated

In locked mode, the Soldeer lane may rewrite only dependencies/ to match the committed metadata. With ALLOW_UPDATE=1, it may update exactly these repository outputs:

dependencies/

The ignored installed dependency tree used by Foundry.

soldeer.lock

The Soldeer lockfile. It records package names, versions, source URLs, and upstream package checksums.

remappings.txt

The import remappings generated for Foundry.

dependency-checksums.txt

The committed local checksum manifest for the installed dependency contents.

What verification proves

In stage mode, scripts/verify-dependencies.sh reads soldeer.lock, downloads each package URL over HTTPS, checks the archive SHA-256 against the lockfile checksum, extracts the archive, confirms the installed dependency tree matches the verified archive payload, and then writes dependency-checksums.txt.

In offline mode, it derives the expected directory name for each dependency and checks that dependencies/ contains exactly those directories. It rejects missing dependency directories, unexpected dependency directories, stray top-level files, duplicate lock entries, missing remappings, stale checksum entries, and checksum mismatches.

The local checksum for each dependency is a deterministic SHA-256 over a sorted manifest of every file path and file hash inside that dependency directory. This is intentionally independent of Soldeer internals. It verifies the installed tree that Foundry will actually read.

Routine commands

# Install only the currently locked Soldeer dependencies.
make deps

# Explicitly allow lock, remapping, and checksum updates.
make deps ALLOW_UPDATE=1

# Check the installed dependency tree without network.
make deps-verify

# Compile or test. These run deps-verify first.
make build
make test
make ci

Adding more Soldeer dependencies

For another Soldeer dependency, add it through the normal Soldeer configuration path, then run make deps ALLOW_UPDATE=1. The command should regenerate soldeer.lock, remappings.txt, dependencies/, and dependency-checksums.txt. Review all committed outputs before committing. 

make deps ALLOW_UPDATE=1
make ci
git diff -- foundry.toml soldeer.lock remappings.txt dependency-checksums.txt

Adding a different dependency source

make deps is allowed to be the umbrella command, but each dependency system should have its own contained lane. Do not put npm, git submodules, code generation, ABI downloads, or other package managers inside the Soldeer lane.

Add a separate stage service.

The networked service reads only its input files and writes only to its own Docker volume.

Add a separate apply service.

The offline apply service writes only the exact repository outputs owned by that dependency source.

Add a separate verify service.

The offline verifier checks the generated state before any build or test consumes it.

Wire it into make deps and make deps-verify.

The public interface can stay simple while the trust boundaries remain explicit.

What this does not prove

This lane proves that the fetched archive matches the committed lockfile checksum and that the installed dependency tree matches the committed local manifest. It does not prove that an upstream package is correct, audited, or safe to use. Reviewing dependency changes is still a human responsibility.
