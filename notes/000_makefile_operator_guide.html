What the Makefile runs

This Makefile is not a shortcut list. It is the operator entrypoint that chooses the container, network, mounts, secrets, and cleanup policy for each kind of work.

The rule

Every target should answer one question before it runs: what authority does this job need? The Makefile maps that answer to a Compose service with a narrow runtime posture.

  • Routine checks build and test offline.
  • Dependency installation is the explicit networked path.
  • Repository writes happen only in named write lanes.
  • RPC, local chains, private keys, and interactive sessions have separate targets.
  • Long-running or stateful services have cleanup targets.

Default privileges

The common posture is non-root, read-only root filesystem, dropped Linux capabilities, no-new-privileges:true, bounded memory/PIDs, and tmpfs caches.

Deviations are intentional: dependency staging gets network, dependency materialization pre-creates guarded host bind targets, ABI generation writes CAM ABI files, JS lanes copy source/manifests into container-local tmpfs and mount node_modules read-only before writing dist/, and local scenario deploys can broadcast to an internal Anvil.

What happens when you run dependency targets

make deps

Solidity dependencies

1. Proxy dependency-egress-proxy is the only service with external network access.
2. Stage soldeer-stage reads only dependency metadata and writes a Docker volume.
3. Verify and patch soldeer-verify-stage verifies upstream archives, applies declared patches, and writes checksums.
4. Apply soldeer-apply-locked or soldeer-apply-update copies staged material into the repo.
5. Verify installed soldeer-verify runs offline against dapps/dependencies/.
Cleanup The Makefile runs docker compose down --volumes --remove-orphans on exit.
The important split is network before repo writes. Soldeer can download into a Docker volume, but the service with network access never receives a writable repository mount.
make package-deps

npm dependencies

Input The Makefile builds a temporary npm project from js/package.json and workspace manifests.
Network package-stage reaches npm only through package-egress-proxy.
Install policy npm lifecycle scripts, audit, funding, and update notifier are disabled.
Apply Offline apply writes js/node_modules/ and, only when requested, js/package-lock.json.
Graph check before use JS build/test/viewer targets run the same offline graph check inside their staged execution workspace before npm scripts.
Guards The target rejects symlinked or wrong-type node_modules, lockfiles, and manifests.
Cleanup Temporary input and Compose volumes are removed on exit.
The npm install process does not see the whole repository. It gets only enough metadata to resolve dependencies, then an offline service materializes the result. Before package scripts run, npm checks that the installed workspace graph is present and consistent with the lockfile. This is not a per-file integrity check of node_modules/.

What happens when you run routine checks

make test

Repository checks and unit tests

First deps-verify checks installed Solidity dependencies.
Then checks runs Python policy tests in compose/checks.yml.
Finally forge-test runs unit tests from compose/forge.yml.
Unit tests are not allowed to bypass repository policy. The Python checks catch source-shape, dependency, import, pragma, and Compose posture problems before Forge unit tests run.
make ci

Default confidence path

Solidity fmt, build, script-build, test, fuzz, and invariant.
Packages package-ci, a Make aggregation over package-test and viewer-terminal-check. The package lane tests the JS workspace; the viewer lane typechecks and smoke-runs the terminal through compose/viewer-terminal.yml.
Network No routine CI lane downloads dependencies or contacts RPC.
ci is a local confidence path over already-materialized dependencies. Dependency updates remain explicit.

What happens when you run local runtime targets

make viewer-terminal

Mock CAM viewer session

Prerequisite Stages JS workspace sources into container-local tmpfs, mounts node_modules read-only, checks the graph, and builds there.
Container viewer-terminal in compose/viewer-terminal.yml.
Tool Runs js/tools/viewer-terminal/terminal-session.ts.
Access No network, read-only js/ input, read-only js/node_modules, read-only shared test fixtures, one read-only mock CAM fixture, staged tmpfs workspace, interactive TTY.
Lifecycle viewer-terminal-status, viewer-terminal-attach, viewer-terminal-down are useful only while the interactive run container still exists.
Check viewer-terminal-check stages JS workspace sources into tmpfs, checks the read-only dependency graph, builds the workspace, typechecks the terminal tool, and runs its entrypoint noninteractively.
Reality Contract reads are mocked; this is not a live-chain viewer.
This is an interactive debugging surface, so it is intentionally networkless and explicit about attach/cleanup.

Live local scenario

Deploy Runs deploy-bike-nft-local after Anvil is healthy.

When adding a new target

Do not add a Make target as a convenience wrapper until its authority is clear. Decide whether it is: an offline check, a networked dependency step, a generated-artifact write, an RPC operation, a local service, or a scenario with secrets.

  • If it needs network, explain which service has egress and why.
  • If it writes the repository, name the exact path and make it generated.
  • If it uses secrets, prefer file-backed secrets and keep raw environment variables out of Compose.
  • If it starts a service, give it a separate Compose project and cleanup target.
  • If its posture matters, add a rendered Compose check under tests/checks/.
